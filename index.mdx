ActiveRecord Multi-Tenant is a Ruby gem that makes it straightforward to build multi-tenant Rails applications on distributed databases like [Citus](https://github.com/citusdata/citus). It automatically scopes your ActiveRecord queries to the current tenant, ensuring data isolation and enabling efficient query routing across database shards.

## Why activerecord-multi-tenant?

When you build a SaaS application on a distributed database like Citus, every query needs to include the tenant partition key so the database can route it to the correct shard. Manually adding `WHERE customer_id = ?` to every query is error-prone and tedious.

This gem solves that by:

- **Automatically injecting the tenant scope** into all SELECT, INSERT, UPDATE, and DELETE statements
- **Providing a clean API** (`MultiTenant.with`, `set_current_tenant`) that works naturally with Rails
- **Integrating with background jobs** via Sidekiq middleware so tenant context is preserved across async work
- **Offering migration helpers** for Citus-specific operations like `create_distributed_table`

## Getting started

<Steps>
  <Step title="Add the gem">
    Add `gem 'activerecord-multi-tenant'` to your Gemfile and run `bundle install`.
  </Step>
  <Step title="Annotate your models">
    Call `multi_tenant :customer` inside each ActiveRecord model that belongs to a tenant.
  </Step>
  <Step title="Set the tenant per request">
    Use `set_current_tenant_through_filter` and a `before_action` in your `ApplicationController` to set the tenant for every request.
  </Step>
  <Step title="Wrap background code">
    Use `MultiTenant.with(tenant) { ... }` anywhere you need to run queries outside of a request context.
  </Step>
</Steps>

<Tip>
  If you are migrating an existing application, use [write-only mode](/concepts/write-only-mode) to roll out tenant scoping incrementally without breaking existing queries.
</Tip>
