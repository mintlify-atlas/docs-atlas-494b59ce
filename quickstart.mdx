---
title: "Quickstart"
description: "Add multi-tenant query scoping to a Rails application in four steps."
---

This guide walks you through the minimum setup needed to have `activerecord-multi-tenant` scoping queries in your Rails application. The example uses a `Customer` tenant and a `Site` model that belongs to a customer.

<Steps>
  <Step title="Install the gem">
    Add the gem to your `Gemfile`:

    ```ruby Gemfile
    gem 'activerecord-multi-tenant'
    ```

    Then install it:

    ```bash
    bundle install
    ```
  </Step>

  <Step title="Annotate your models">
    Call `multi_tenant` with the name of your tenant model inside every ActiveRecord model that belongs to a tenant. Use the singular, underscored name of the tenant model as the argument.

    ```ruby app/models/site.rb
    class Site < ActiveRecord::Base
      multi_tenant :customer
      has_many :page_views

      # ...
    end
    ```

    ```ruby app/models/page_view.rb
    class PageView < ActiveRecord::Base
      multi_tenant :customer
      belongs_to :site

      # ...
    end
    ```

    Once a model declares `multi_tenant :customer`, the gem:

    - Adds a `customer_id` partition key to the model.
    - Automatically stamps new records with the current tenant's ID.
    - Appends `WHERE customer_id = ?` to all queries when a tenant is set.
    - Raises `MultiTenant::TenantIsImmutable` if you attempt to change the tenant of a persisted record.
  </Step>

  <Step title="Set the tenant per request">
    In your `ApplicationController`, opt in to the filter-based tenant helper and set the current tenant in a `before_action`:

    ```ruby app/controllers/application_controller.rb
    class ApplicationController < ActionController::Base
      set_current_tenant_through_filter # required to opt in
      before_action :set_customer_as_tenant

      private

      def set_customer_as_tenant
        customer = Customer.find(session[:current_customer_id])
        set_current_tenant(customer)
      end
    end
    ```

    After this, every request automatically has the correct tenant scope. All queries in controllers, views, and helpers will be filtered to the current customer.
  </Step>

  <Step title="Scope code outside of requests">
    For code that runs outside of a web request — such as Rake tasks, scripts, or background jobs — use `MultiTenant.with` to set the tenant for the duration of a block:

    ```ruby
    customer = Customer.find(session[:current_customer_id])

    MultiTenant.with(customer) do
      site = Site.find(params[:site_id])
      site.update!(last_accessed_at: Time.now)
      site.page_views.count
    end
    ```

    You can also pass a raw ID instead of a model instance:

    ```ruby
    MultiTenant.with(42) do
      Site.all # => SELECT * FROM sites WHERE customer_id = 42
    end
    ```
  </Step>
</Steps>

<Note>
  **Migrating an existing application?** If your database already has rows without a `customer_id`, enable write-only mode first. In this mode the gem stamps the tenant ID on new records but does not add the tenant scope to reads, giving you time to backfill existing rows.

  ```ruby config/initializers/multi_tenant.rb
  MultiTenant.enable_write_only_mode
  ```

  Once all rows have a `customer_id` and you have made the column `NOT NULL`, remove that line. See the [rolling out guide](/guides/rolling-out) for the full migration strategy.
</Note>
